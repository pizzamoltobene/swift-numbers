# SwiftNumbers Growth Campaign Playbook (Q2 2026)

Status: ready to execute  
Owner: Growth Lead  
Project: `swift-numbers`

## 1) Campaign Goal

Primary goal for next 30 days:

- Increase qualified awareness among Swift/iOS developers.
- Convert awareness into package installs, GitHub stars, and active feedback.

Suggested targets for the first 30-day sprint:

- +300 GitHub stars
- +10 external mentions (newsletters, forums, posts)
- +20 qualified inbound issues/discussions from real users
- +5 production use-case conversations

## 2) Positioning (Core Message)

Use one clear line everywhere:

- "SwiftNumbers is a Swift-native library and CLI to read and edit Apple .numbers files safely."

Proof points pulled from the current repo:

- Native Swift stack (SwiftPM)
- Real `.numbers` read/edit/save workflows
- CLI for automation (`list-sheets`, `list-tables`, `read-cell`, `read-range`, `export-csv`, `import-csv`, `dump`)
- Strong docs surface (quickstart, capabilities, API reference, cookbook)

## 3) Roles and Ownership

Keep ownership explicit. One person may hold multiple roles.

1. Growth Lead (human)
- Owns weekly priorities and channel budget.
- Final approver for outbound messaging.

2. PMM Operator (Codex)
- Creates message variants, value props, and launch narrative.
- Maintains campaign docs and templates.

3. Content Operator (Codex + human review)
- Produces X posts, threads, demos, and technical snippets.
- Ships weekly content batches.

4. Distribution Operator (human + Codex support)
- Submits to newsletters/communities.
- Tracks outreach and follow-ups.

5. Analytics Operator (Codex)
- Maintains weekly dashboard.
- Recommends stop/keep/scale decisions.

## 5) Channel Playbooks

## X (primary always-on channel)

Cadence:

- Daily: 1 post
- Weekly: 2 threads
- Weekly: 1 mini-demo video

Content ratio:

- 40% practical demos/tutorial snippets
- 30% behind-the-scenes build notes
- 20% user outcomes and testimonials
- 10% direct CTA posts

CTA examples:

- "Try this on your own `.numbers` file."
- "Reply with a failing `.numbers` edge case and we will test it."
- "Star/watch the repo if this saves you spreadsheet pain."

## Swift Forums

Post in Community Showcase with:

- Problem statement
- What SwiftNumbers does today (no hype)
- Concrete example command/API
- Known limitations (build trust)
- Request for real edge-case files and feedback

## Swift Package Index

Checklist:

1. Package is listed.
2. Semver tags are present and current.
3. Package metadata and docs links are complete.
4. DocC hosting setup is enabled if applicable.

## Newsletters

Use two modes:

1. Organic submissions (free)
2. Paid sponsorship (for defined launch windows only)

Do not buy sponsorship before message/asset testing on X and forums.

## 6) Weekly KPI Dashboard

Track every Monday:

1. Reach
- X impressions
- GitHub repository views

2. Activation
- SwiftPM installs/clones proxy (if available)
- New stars/watchers
- Docs page views

3. Engagement quality
- Qualified inbound issues/discussions
- External mentions
- Demo requests

4. Conversion proxy
- "Used in project" confirmations
- Repeat contributors

## 7) Operating Rhythm (War Room)

Run this every week:

1. Monday (30 min): plan priorities and experiments.
2. Tuesday-Thursday: execute and respond quickly.
3. Friday (30 min): metrics review and next-week decisions.

Decision rule:

- Keep if channel has qualified responses.
- Kill if channel has noise only.
- Scale if channel shows repeat qualified engagement.

## 8) Guardrails

1. No fake engagement or synthetic social proof.
2. No ban evasion behavior on any platform.
3. No exaggerated capability claims.
4. Be explicit about scope and limitations.
